# 已知问题与决策

本文档记录实际封装/还原过程中遇到的问题，以及当前项目采用的处理策略。

## 1. 文档和脚本说明语言

决策：

- 文档、脚本注释、用户可见日志尽量使用中文。
- schema 字段名、manifest 键名、路径变量名保留英文。
- PowerShell 函数名、参数名保持英文，便于和系统命令、编辑器补全配合。

## 2. NAS 目录策略

决策：

- 本项目优先使用英文自包含目录：`\\192.168.1.37\backups\win11-image-kit`
- `/data1/下载` 仍作为日常下载池，安装包下载完成后再归集到项目目录。
- `/data1/work/win11-image-kit` 用作临时工作区。
- 旧的 `\\192.168.1.37\backups\packages` 是早期规划产物，本项目后续不再使用。

## 3. Codex 安装包

Codex 纳入软件安装清单。默认安装包来源：

```text
\\192.168.1.37\backups\win11-image-kit\packages\ai-tools\Codex Installer.exe
```

如果实际文件名或目录不同，只调整 `manifests/software.json` 中的 `source`。

## 4. 便携版 EXE 被 Windows 安全中心隔离

部分便携版工具、绿色版工具或压缩包中的单文件 EXE 可能被 Windows 安全中心误判，部署到新机后被隔离或删除。

当前方案：

- 不关闭 Microsoft Defender。
- 不做全盘排除。
- 不排除整个下载目录。
- 通过 manifest 声明“明确需要保留”的工具目录或进程。
- 部署后用脚本批量添加排除项。

相关文件：

```text
manifests\defender-exclusions.json
scripts\postdeploy\Set-DefenderExclusions.ps1
```

## 5. 百度网盘还原后消失

现象：

新机还原镜像后，百度网盘相关文件彻底消失。初步怀疑和 Sysprep 前 AppX 清理有关。

可能原因：

- 百度网盘安装器可能注册了 per-user AppX/Desktop Bridge 包。
- 清理 AppX 时如果按包名误删，可能连同用户态应用注册和文件一起清除。
- 某些软件本体安装在用户目录、WindowsApps 或 per-user package 目录中，不像传统 `Program Files` 软件那样稳定保留。

当前策略：

1. Sysprep 前先执行包名审计，不直接盲删。
2. 对百度网盘这类需要保留的软件，先记录包名，再加入保留列表。
3. 如果仍不稳定，则把百度网盘改为“部署后安装”，不要固化进镜像。
4. 部署后阶段用安装包自动或半自动恢复。

建议排查命令：

```powershell
Get-AppxPackage -AllUsers | Where-Object {
    $_.Name -match "Baidu|Netdisk|3AD8A913"
} | Select-Object Name, PackageFullName, InstallLocation, PackageUserInformation
```

## 6. AppX 清理原则

决策：

- `Clear-AppxForSysprep.ps1` 只处理明确不需要的 Microsoft 预装应用。
- 第三方应用不默认删除。
- 后续需要把 AppX 包名、处理动作、是否保留写入 manifest，避免脚本里硬编码越来越多。

## 7. 定制范围开放给用户修改

决策：

- 项目的选择权放在 manifest 中，脚本只按 manifest 执行。
- `manifests/customization-scope.json` 是总定制入口。
- `manifests/paths.json` 是工作目录和根路径入口。
- 用户可以在启动构建前增删 `software.json` 中的软件项，也可以把 `enabled` 改为 `false` 临时停用。
- 系统项、AppX 清理、Defender 排除项、火绒安装策略都应通过 manifest 表达，不应散落在脚本里硬编码。

## 8. Decision: UX restore must be version-aware and scope-aware

决策：

- 默认应用、开始菜单和任务栏恢复必须记录来源 Windows edition/version/build、目标作用域、模板类型和目标应用能力。
- `default-user` 不等于 `current-user`，`offline-image` 不等于当前运行机器。
- 写入 Default Profile 不等于当前用户已修改。
- 不把 DISM、复制命令或其它命令的退出码当作“用户体验已生效”的证据。
- 不使用依赖内部数据库或高风险注册表 hack 的固定方案。
- 版本不支持、作用域不匹配、目标应用缺失或 ProgId unknown 时，必须进入报告并阻断后续真实执行。
- 当前 Issue #18 report-only / fixture / plan-only 阶段已有 post-PR #96 main/workflow ready evidence；人工关闭交接记录在 docs/64，真实恢复必须由 docs/65 所述后续任务单独授权并提供验证证据。

## Decision 9: Restore-UserExperience 入口先迁移为 handler report，不执行恢复

- 状态：accepted
- 背景：旧入口可能把 Default Profile、命令退出码或预演结果误读为当前用户体验已恢复。
- 决策：`Restore-UserExperience.ps1` 当前只输出 `restore-user-experience` handler report，读取 `configs/default-apps` 与 `configs/start-menu` metadata，生成 planned/manual/blocked 和 manual checklist。`-Apply` / `-Execute` 在本阶段必须 blocked。
- 结果：postdeploy 汇总仍保留 `userExperienceSummary` 兼容字段，但真实恢复、registry/profile/default app/Start menu/taskbar mutation 和真实 evidence collection 留给后续授权任务。Issue #18 当前 report-only / handler-adapter stage 已有 post-PR #96 main/workflow ready evidence，可进入人工收口评估；docs/64 是 handoff，不是 completion summary。
