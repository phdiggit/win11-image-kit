# 部署后恢复

目标：新机还原后，除账号登录和授权激活外，其余配置尽量自动完成。

## 执行时机

OOBE 完成、进入桌面、网络可用、Windows 初始驱动安装稳定后执行。

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
.\scripts\postdeploy\Invoke-PostDeploy.ps1
```

如需先预演会执行哪些危险动作，使用：

```powershell
.\scripts\postdeploy\Invoke-PostDeploy.ps1 -WhatIf
```

`-WhatIf` 预演默认允许在非管理员会话中执行；脚本会自动降级为只输出计划、日志和轻量报告，不做真实系统写入。

如需只预演用户体验恢复步骤，使用：

```powershell
.\scripts\postdeploy\Restore-UserExperience.ps1 -WhatIf
```

## 自动恢复内容

- 按 `manifests/paths.json` 解析项目路径、工具目录和数据目录
- 按 `manifests/customization-scope.json` 确认本轮定制范围
- Windows Defender 排除项
- D 盘数据重定向
- 默认应用关联：如果 `DefaultAppAssoc.xml` 存在，则通过 DISM 导入
- 资源管理器选项：显示扩展名、显示隐藏文件等当前用户选项
- 开始菜单默认布局：如果 `LayoutModification.json` 存在，则复制到默认用户配置，主要影响后续新建用户
- Windows Terminal 配置模板：如果 `${ConfigRoot}\windows-terminal` 存在且非空，则恢复到当前用户的 Terminal `LocalState`
- VSCode 便携版 data 配置目录：优先从 `${ConfigRoot}\vscode-portable\data` 恢复；模板缺失时只告警，并检查或准备 `${ToolRoot}\vscode-portable\data`
- 部署后软件：处理 `software.json` 中 `stage=post-deploy` 的归档包和可静默安装器；非静默安装器永不自动执行，只保留为人工项
- 数据库/中间件服务注册：执行 `manifests/services.json` 中的 `install` 命令，已存在服务会跳过，不自动启动服务；注册后会按服务清单中的 `expectedState`、`expectedStartType`、`required` 和 `failurePolicy` 生成结构化状态验证结果

普通模式下，缺失用户体验模板只会输出 `WARN` 和人工建议，不会导致部署失败。只有显式启用严格模式时，缺失模板才会终止用户体验恢复步骤。

## 用户体验恢复计划模式（Issue #18）

Issue #18 将默认应用、开始菜单和任务栏恢复推进为 version-aware、scope-aware 的 report-only planning。当前阶段只生成 capability matrix、template metadata、scope semantics 和 verification plan，不实际导入默认应用、不写 Default Profile、不修改当前用户开始菜单或任务栏。

`Restore-UserExperience.ps1` 已接入 scope-aware handler report，默认 `-WhatIf` / `plan-only`。默认应用与开始菜单 metadata 位于 `configs/default-apps/` 和 `configs/start-menu/`；这些 metadata 只用于计划、阻断和人工清单。`-Apply` / `-Execute` 当前会被 blocked，不代表真实授权。

Issue #18 当前 report-only / handler-adapter stage 已有 post-PR #96 main/workflow ready evidence，可进入人工收口评估。人工关闭交接见 [Issue #18 Manual Closure Handoff](64-issue18-manual-closure-handoff.md)，真实执行拆分见 [Future True UX Restore Execution Split](65-future-true-ux-restore-execution-split.md)。PR Fast CI、fixture、handler report 和 manual checklist 只能作为 report-only 安全边界验证，不能证明真实用户体验已恢复。

`default-user`、`current-user`、`offline-image` 和 `machine` 是不同作用域。写入 Default Profile 不等于当前用户已变更；离线镜像计划也不等于当前运行系统已变更。命令退出码只能说明命令本身完成，不足以证明用户体验已经恢复。真实操作需要后续明确授权、受支持的执行方式和可验证证据。

如需输出部署后安装器计划与执行结果，可指定 `-ReportPath`：

```powershell
.\scripts\postdeploy\Invoke-PostDeploy.ps1 -WhatIf -ReportPath .\logs\postdeploy-installer-plan.json
```

如需输出总日志和总报告，可指定 `-LogPath` 与 `-SummaryReportPath`。如需输出服务状态验证报告，可指定 `-ServiceReportPath`。如需输出用户体验恢复报告，可指定 `-UserExperienceReportPath`；如需把缺失模板视为失败，可再加 `-StrictUserExperience`：

```powershell
.\scripts\postdeploy\Invoke-PostDeploy.ps1 -WhatIf `
  -LogPath .\logs\postdeploy.log `
  -SummaryReportPath .\logs\postdeploy-summary.md `
  -ReportPath .\logs\postdeploy-installer-plan.json `
  -ServiceReportPath .\logs\postdeploy-services.json `
  -UserExperienceReportPath .\logs\postdeploy-user-experience.json

.\scripts\postdeploy\Restore-UserExperience.ps1 `
  -Strict `
  -ReportPath .\logs\restore-user-experience.md
```

如在 `customization-scope.json` 中启用了 `reporting.postDeploy`，则不传这些路径时也会自动归档到 `${DeployRoot}\logs\postdeploy` 和 `${DeployRoot}\reports\postdeploy`；NAS 不可写时会给出降级说明并继续控制台输出。

部署后总报告会在 JSON 和 Markdown 中写入统一的 `childReportSummary`。该字段汇总安装器、服务、Junction、Defender 和用户体验子报告的总数、存在数、缺失数、`failedRequired`、`failedOptional`、`hasBlockingFailure` 和建议 `exitCode`。子报告引用只保留各自的 compact summary 与路径、错误信息，不把完整 `packageResults`、`serviceResults`、`junctionResults`、`defenderResults` 或 `userExperienceResults` 数组内嵌到顶层报告；required 子报告缺失、解析失败或包含 required 失败时，会让顶层 `childReportSummary.hasBlockingFailure` 标记为 `true`。
