# 定制范围与配置入口

本项目的最终目标是：定制 Windows 系统镜像，让镜像还原到新机后只需要做最少的手工操作。理想的手工操作只包括账号登录、授权激活、硬件相关确认，以及少量必须由人判断的安全确认。

## 配置入口

用户启动本项目后，优先修改这些 manifest，而不是直接改脚本：

| 文件 | 用途 |
|---|---|
| `manifests/customization-scope.json` | 总定制范围，决定系统项、应用项、AppX 清理和部署阶段是否启用 |
| `manifests/paths.json` | NAS、本机工具目录、数据目录、临时工作区等路径 |
| `manifests/software.json` | 用户应用、便携包、安装器和后续验证命令 |
| `manifests/appx-cleanup.json` | 要卸载或保留的系统 AppX 应用 |
| `manifests/defender-exclusions.json` | Windows Defender 可信目录和进程排除项 |
| `manifests/junctions.json` | 需要迁移到数据盘并建立 Junction 的目录 |
| `manifests/services.json` | 数据库、中间件等服务的注册、停止和删除命令 |

其中 `customization-scope.json` 是总入口。它不复制所有细节，只声明当前 profile 要启用哪些定制模块，以及每个模块使用哪个 manifest。

## 系统相关定制

系统相关定制应覆盖：

- 右键菜单，例如“通过 Code 打开”。
- 开始菜单和任务栏固定项。
- Windows Terminal 配置模板。
- 资源管理器选项，例如显示扩展名、显示隐藏文件。
- 默认应用关联。
- Windows Defender 策略：保持开启、开启但添加排除项、禁用。
- 是否安装火绒。

策略：

- 默认不禁用 Windows Defender。
- 如必须保留高误报便携工具，优先使用 `defender-exclusions.json` 精确排除。
- 是否安装火绒由 `customization-scope.json` 中的 `system.huorong.install` 控制；安装包仍应在 `software.json` 中声明。

## 应用相关定制

应用相关定制分三类：

- 系统应用：通过 `appx-cleanup.json` 决定卸载或保留。
- 系统内置应用配置：例如 Windows Terminal、默认应用、开始菜单。
- 用户配置目录恢复：例如 VSCode 便携版 `data` 目录。
- 用户应用：通过 `software.json` 增删安装器或便携包。

其中，`customization-scope.json` 中的 `system.windowsTerminal` 和 `system.vscodePortable` 用于声明部署后恢复所需的模板来源和目标目录；脚本默认宽松处理，模板缺失时只告警，显式启用严格模式才会失败。

如需让构建、部署后恢复或配置验证自动归档轻量日志和报告，可在 `customization-scope.json` 的 `reporting.build`、`reporting.postDeploy`、`reporting.validation` 中分别启用，并把目录指向 `${DeployRoot}\logs\...` 与 `${DeployRoot}\reports\...`。默认关闭，因此日常验证不会强制访问 NAS。

`software.json` 中每个软件包可以使用：

- `enabled`：是否启用。
- `version`：软件版本，未知或滚动安装器可先写 `latest`，但不要省略。
- `category`：分类，例如 `editor`、`ai-tools`、`dev-runtime`。
- `stage`：执行阶段，取值为 `golden-image`、`post-deploy`、`manual`，必须显式声明。
- `type`：安装介质类型，例如 `archive`、`installer`、`manual`。
- `archiveFormat`：归档格式，例如 `zip`、`tar.gz`。
- `sha256`：安装介质的 SHA256；暂时未知时可写 `null`，归档安装和 `-CheckPackageFiles` 会在有值时校验。
- `installArgs`：安装器参数数组，供后续静默安装 handler 使用。
- `silentInstall`：安装器是否具备可重复静默安装能力。
- `successExitCodes`：安装器允许视为成功的退出码数组；未声明时默认只接受 `0`。
- `uninstall`：卸载命令，未知时可写 `null`。
- `postInstall`：结构化后置动作，例如 `ensure-directory`，不要写任意 PowerShell 字符串。

用户可以提前把安装程序或便携包放到 `paths.json` 中 `PackageRoot` 指向的位置，也可以启动项目后再把软件项加入 `software.json`。删除或临时停用软件时，优先把 `enabled` 改为 `false`，不要急着删除记录。

## 工作目录配置

所有项目路径集中在 `manifests/paths.json`：

- `NasProjectRoot`：NAS 项目根目录。
- `PackageRoot`：安装包和便携包归集目录。
- `ConfigRoot`：可版本化配置导出目录。
- `ImageRoot`：Windows 镜像目录。
- `DeployRoot`：部署脚本、日志、报告目录。
- `WorkRoot`：SSD 临时工作区。
- `DownloadPool`：日常下载池。
- `ToolRoot`：新机工具安装根目录。
- `DataRoot`：新机数据根目录。

脚本和 manifest 中可以引用 `${PackageRoot}`、`${ToolRoot}`、`${DataRoot}` 等变量。换 NAS、换 D 盘目录或换工具安装根目录时，应优先修改 `paths.json`。

## 推荐交互流程

1. 编辑 `paths.json`，确认 NAS、本机工具目录和数据目录。
2. 编辑 `customization-scope.json`，决定这轮镜像要启用哪些系统和应用定制。
3. 编辑 `software.json`，增删用户应用、安装器和便携包。
4. 编辑 `appx-cleanup.json`，确认只删除明确不需要的 Microsoft 预装应用。
5. 编辑 `defender-exclusions.json`，只为明确可信的便携工具添加排除项。
6. 执行 `scripts/config/Show-CustomizationScope.ps1`，检查当前 profile 和路径解析结果。
7. 执行 `scripts/validate/Test-ProjectConfig.ps1`，检查 manifest 引用、路径变量和脚本语法。
8. 执行构建、封装前检查、部署后恢复和测试脚本。

这套流程的原则是：选择权放在 manifest，执行逻辑放在脚本。用户可以随时改定制范围，脚本只负责按当前 manifest 执行和记录结果。
